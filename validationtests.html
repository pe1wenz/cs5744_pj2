Validation Tests for SchedRunner

1. Purpose

The validation tests are designed to confirm that the SchedRunner system aligns with the specified functional, performance, and internal design requirements. These tests ensure the complete system meets its end-to-end workflow requirements as outlined in the project scope. This section includes the creation of a traceability matrix to map test cases to system requirements, providing a systematic approach to verifying compliance.

2. Validation Test Plan

2.1 Test Objectives
  • Validate that all functional requirements (FRs) are met, including job configuration, dependency management, scheduling, and real-time monitoring.
  • Ensure the system satisfies non-functional requirements (NFRs) such as performance under load, scalability, security, and fault tolerance.
  • Confirm system integrity through comprehensive logging and secure communication.
2.2 Test Environment
  • A simulated database for test jobs and dependencies.
  • Mock systems for worker nodes and target machines to replicate real-world conditions.
  • Performance monitoring tools to measure response times, scalability, and fault tolerance.
  • API testing tools for validating secure interactions and CRUD operations.
2.3 Software Tools
  • Traceability Tool: Maps test cases to system requirements.
  • Simulation Frameworks: Dependency simulators and mock APIs.
  • Load Testing Tools: Simulate high job volumes and monitor performance.
  • Monitoring Dashboard: Validate real-time updates and user interactions.
  • Error Injection Tools: Test system responses to faults and failures.

3. Validation Test Cases

3.1 Functional Validation
  • Job Creation:
    • Create jobs with detailed descriptions, schedules, and dependencies.
    • Validate that all job attributes are stored correctly in the database.
  • Job Dependency Management:
    • Simulate completion of dependencies and confirm job state transitions (e.g., "Waiting" â†’ "Pending").
  • Real-Time Monitoring:
    • Verify that job statuses (e.g., "Pending," "Running," "Failed") are updated in the dashboard within 2 seconds.
  • User Actions:
    • Test user-triggered operations such as starting, pausing, and terminating jobs.
    • Confirm that job states are reflected immediately in the UI.
  • Comprehensive Logging:
    • Validate that all job-related events (creation, execution, errors) are logged with accurate timestamps.
3.2 Performance Validation
  • Scalability:
    • Add and remove worker nodes dynamically and monitor job redistribution.
    • Test the system under high job volumes and ensure load balancing.
  • Response Time:
    • Verify response times for user-triggered actions and real-time updates are within specified limits.
  • Fault Tolerance:
    • Simulate node failures and test the system's failover mechanisms.
    • Validate state recovery without data loss.
3.3 Security Validation
  • API Authentication:
    • Test OAuth token handling and Active Directory integration for secure API interactions.
    • Verify error logging for unauthorized access attempts.
  • Data Security:
    • Ensure encrypted communication between system components to maintain data confidentiality and integrity.
3.4 End-to-End Validation
  • Simulate a complete workflow:
    • Create a job with dependencies and scheduling requirements.
    • Monitor the job's execution across target machines.
    • Verify the final state and ensure logs capture the full lifecycle of the job.

4. Results Evaluation

  • Pass Criteria: The test meets all functional and non-functional requirements.
  • Fail Criteria: The test fails to meet one or more requirements, requiring re-evaluation of the system design or implementation.

FR-1 Job creation with detailed configurations Verify user ability to create jobs with descriptions, schedules, and dependency definitions.
